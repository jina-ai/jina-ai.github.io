Уншигчидтай холбоотой нийтлэг асуултууд
Reader API ашиглахтай холбоотой ямар зардал гарах вэ?
Reader API нь үнэ төлбөргүй бөгөөд API түлхүүр шаарддаггүй. Зүгээр л өөрийн URL дээр 'https://r.jina.ai/' гэж бичнэ үү.
Reader API хэрхэн ажилладаг вэ?
Reader API нь прокси ашиглан дурын URL-г дуудаж, агуулгыг нь вэб хөтөч дээр буулгаж, өндөр чанартай үндсэн контентыг гаргаж авдаг.
Reader API нээлттэй эх сурвалж мөн үү?
Тийм ээ, Reader API нь нээлттэй эх сурвалж бөгөөд Jina AI GitHub репозитор дээр байдаг.
Reader API-ийн ердийн хоцролт гэж юу вэ?
Reader API нь ерөнхийдөө URL-уудыг боловсруулж, 2 секундын дотор агуулгыг буцаана, гэхдээ нарийн төвөгтэй эсвэл динамик хуудсууд илүү их цаг шаардаж магадгүй юм.
Би яагаад хуудсыг өөрөө хусахын оронд Reader API ашиглах ёстой вэ?
Ялангуяа нарийн төвөгтэй эсвэл динамик хуудсуудыг хусах нь төвөгтэй бөгөөд найдваргүй байж болно. Reader API нь цэвэр, LLM-д бэлэн текстийн оновчтой, найдвартай гаралтыг өгдөг.
Reader API нь олон хэлийг дэмждэг үү?
Reader API нь контентыг URL-ийн эх хэлээр буцаана. Энэ нь орчуулгын үйлчилгээ үзүүлдэггүй.
Хэрэв вэбсайт Reader API-г хаасан бол би яах ёстой вэ?
Хэрэв танд блоклох асуудал тулгарвал манай тусламжийн багтай холбогдож тусламж, шийдвэрлэнэ үү.
Reader API нь PDF файлаас контент гаргаж чадах уу?
Үндсэндээ вэб хуудсуудад зориулагдсан боловч Reader API нь arXiv гэх мэт вэб сайтууд дээр HTML форматаар үзсэн PDF-ээс контент гаргаж авах боломжтой боловч ерөнхий PDF задлахад оновчтой биш юм.
Reader API нь вэб хуудаснаас медиа контентыг боловсруулж чадах уу?
Одоогоор Reader API нь медиа контентыг боловсруулдаггүй ч ирээдүйн сайжруулалтад зургийн тайлбар, видео хураангуй орно.
Орон нутгийн HTML файлууд дээр Reader API ашиглах боломжтой юу?
Үгүй ээ, Reader API нь зөвхөн нийтэд нээлттэй URL-уудын контентыг боловсруулах боломжтой.
Reader API контентыг кэш хийдэг үү?
Хэрэв та ижил URL хаягийг 5 минутын дотор хүсэх юм бол Reader API нь кэштэй контентыг буцаана.
Би нэвтрэлтийн цаана байгаа контентод хандахын тулд Reader API ашиглаж болох уу?
Харамсалтай нь үгүй.
Би arXiv дээр PDF руу хандахын тулд Reader API ашиглаж болох уу?
Тийм ээ, та Уншигчаас (https://r.jina.ai/https://arxiv.org/pdf/2310.19923v4) уугуул PDF дэмжлэгийг ашиглах эсвэл arXiv (https://)-ийн HTML хувилбарыг ашиглаж болно. r.jina.ai/https://arxiv.org/html/2310.19923v4)
Reader дээр зургийн тайлбар хэрхэн ажилладаг вэ?
Уншигч заасан URL дээрх бүх зургийн тайлбарыг бичээд `Image [idx]: [ caption]`-г alt шошго болгон нэмнэ (хэрэв тэдгээрт анх байхгүй бол). Энэ нь доод түвшний LLM-д тайлбар хийх, нэгтгэн дүгнэх гэх мэт зургуудтай харилцах боломжийг олгодог.
Уншигчийн өргөтгөх чадвар юу вэ? Би үүнийг үйлдвэрлэлд ашиглаж болох уу?
Reader API нь маш томруулж болохуйцаар бүтээгдсэн. Энэ нь бодит цагийн траффик дээр тулгуурлан автоматаар хэмжигддэг бөгөөд хамгийн дээд зэрэглэлийн хүсэлт нь одоо 4000 орчим байна. Бид үүнийг Jina AI-ийн үндсэн бүтээгдэхүүний нэг болгон идэвхтэй хадгалж байна. Тиймээс үйлдвэрлэлд чөлөөтэй ашиглах боломжтой.
Reader API-ийн хурдны хязгаар хэд вэ?
Доорх хүснэгтээс ханшийн хязгаарын хамгийн сүүлийн үеийн мэдээллийг авна уу. Бид Reader API-ийн хурдны хязгаар болон гүйцэтгэлийг сайжруулахаар идэвхтэй ажиллаж байгаа тул хүснэгтийг зохих ёсоор шинэчлэх болно гэдгийг анхаарна уу.

Reranker API хэр үнэтэй вэ?
Reranker API-ийн үнэ нь манай Embedding API-ийн үнийн бүтэцтэй нийцдэг. Энэ нь шинэ API түлхүүр бүрт 1 сая үнэгүй жетоноор эхэлдэг. Үнэгүй жетоноос гадна өөр өөр багцуудыг худалдан авах боломжтой. Дэлгэрэнгүй мэдээллийг манай үнийн хэсгээс авна уу.
Хоёр дахин эрэмбэлэгчийн хооронд ямар ялгаа байна вэ?
Гол ялгаа нь тэдний архитектурт оршдог. Гүйцэтгэлийн хувьд бид jina-reranker-v1-ийг санал болгож байна, үүнийг өргөнөөр туршиж, өрсөлдөгчидтэй харьцуулсан. Jina-reranker-v1 нь cross-encoder архитектурыг ашигладаг бол Jina-colbert-v1 нь ColBERTv2 архитектур дээр суурилсан боловч асуулга болон баримт бичгийн аль алиных нь контекстийн уртыг 8192 болгон өргөтгөж, анхны ColBERTv2 загвараас ч илүү сайн гүйцэтгэлтэй болсон.
Jina Reranker нээлттэй эх сурвалж мөн үү?
Тиймээ, jina-colbert-v1 нь нээлттэй эх сурвалж бөгөөд Huggingface-ээр дамжуулан хандах боломжтой. Гэсэн хэдий ч jina-reranker-v1 нь нээлттэй эх сурвалж биш юм.
Дахин эрэмбэлэгч олон хэлийг дэмждэг үү?
Одоогоор зөвхөн англи хэлийг дэмждэг. Гэсэн хэдий ч зарим хэрэглэгчид Хятад хэлтэй ч сайн ажилладаг гэж мэдээлсэн. Энэ нь зарим талаараа jina-reranker-v1-base-en нь манай jina-embeddings-v2-base-zh шигтгэх загвартай зарим нэг жинг хуваалцдагтай холбоотой байж болох юм.
Асуулт, баримт бичгийн дээд хэмжээ хэд вэ?
Асуулгын хамгийн их урт нь 512. Баримт бичигт токены хязгаарлалт байхгүй.
Нэг асуулгад дахин эрэмбэлэх баримт бичгийн дээд хэмжээ хэд вэ?
Та асуулга бүрт 2048 хүртэлх баримт бичгийг дахин эрэмбэлэх боломжтой.
Багцын хэмжээ хэд вэ, би нэг хүсэлтэд хэдэн баримт бичиг илгээх боломжтой вэ?
Манай Embedding API-ээс ялгаатай нь багцын хэмжээ гэсэн ойлголт байдаггүй. Та хүсэлт бүрд зөвхөн нэг асуулга-баримт бичгийн багц илгээх боломжтой боловч уг багцад 2048 хүртэлх нэр дэвшигчийн баримт бичиг багтах боломжтой.
100 баримтыг дахин эрэмбэлэх үед би ямар хоцрогдол хүлээж болох вэ?
Хоцролт нь баримт бичгийн урт, асуулга зэргээс шалтгаалан 100 миллисекундээс 7 секундын хооронд хэлбэлздэг. Жишээлбэл, тус бүр нь 256 жетон бүхий 100 баримтыг 64 жетон асуулгатай дахин эрэмбэлэх нь ойролцоогоор 150 миллисекунд болдог. Баримт бичгийн уртыг 4096 токен болгон нэмэгдүүлснээр энэ хугацаа 3.5 секунд болж нэмэгддэг. Хэрэв асуулгын уртыг 512 токен болгон нэмэгдүүлбэл хугацаа 7 секунд хүртэл нэмэгдэнэ.
Нэг асуулга, 100 баримтыг миллисекундээр дахин эрэмбэлэх цаг хугацааны зардлыг доор харуулав.

Jina-clip-v1 ашиглах үед токенуудыг хэрхэн тооцдог вэ?
Токенуудыг текстийн урт болон зургийн хэмжээнээс хамаарч тооцдог. Хүсэлт дэх текстийн хувьд жетоныг стандарт аргаар тооцдог. Хүсэлт дэх зургийн хувьд дараах алхмуудыг гүйцэтгэнэ. 1. Хавтангийн хэмжээ: Зураг бүрийг 224x224 пиксел хэмжээтэй хавтан болгон хуваасан. 2. Хамрах хүрээ: Оруулсан зургийг бүрэн хамрахад шаардагдах хавтангийн тоог тооцоолно. Зургийн хэмжээсүүд нь 224-т бүрэн хуваагддаггүй байсан ч бид хэсэгчилсэн хавтанг бүрэн хавтан гэж тооцно. 3. Нийт хавтан: Зургийг бүрхсэн хавтангийн нийт тоо нь зардлыг тодорхойлдог. Жишээлбэл, хэрэв зураг нь 500x500 пиксел бол 3х3 хавтангаар бүрхэгдсэн бөгөөд 9 хавтантай болно. 4. Зардлын тооцоо: Хавтан бүр нь зургийг боловсруулах эцсийн зардалд хувь нэмэр оруулдаг. Нэг хавтангийн үнэ 1000 жетон байна. Жишээ: 500x500 пикселийн хэмжээтэй зургийн хувьд: • Зургийг 224x224 пикселийн хэмжээтэй хавтан болгон хуваасан. • Шаардлагатай нийт хавтангийн тоо 3 (хэвтээ) x 3 (босоо) = 9 хавтан байна. • Зардал нь 9*1000 = 9000 жетон байх болно

Сегменттэй холбоотой нийтлэг асуултууд
Segmenter API хэр үнэтэй вэ?
Segmenter API ашиглахад үнэ төлбөргүй байдаг. API түлхүүрээ өгснөөр та илүү өндөр тарифын хязгаарт хандах боломжтой бөгөөд таны түлхүүрийг төлөхгүй.
Хэрэв би API түлхүүр өгөхгүй бол ханшийн хязгаар хэд вэ?
API түлхүүргүйгээр та Segmenter API-д 20 RPM хурдны хязгаарт хандах боломжтой.
Хэрэв би API түлхүүр өгвөл тарифын хязгаар хэд вэ?
API түлхүүрээр та Segmenter API-д 200 RPM хурдны хязгаарт хандах боломжтой. Дээд зэрэглэлийн төлбөртэй хэрэглэгчдийн хувьд хурдны хязгаар нь 1000 RPM байна.
Та миний API түлхүүрээс токенуудыг цэнэглэх үү?
Үгүй ээ, таны API түлхүүрийг зөвхөн дээд түвшний хязгаарт хандахад ашигладаг.
Segmenter API нь олон хэлийг дэмждэг үү?
Тиймээ, Segmenter API нь олон хэлтэй бөгөөд 100 гаруй хэлийг дэмждэг.
GET болон POST хүсэлтүүдийн хооронд ямар ялгаа байдаг вэ?
GET хүсэлтийг зөвхөн текст дэх токенуудын тоог тоолоход ашигладаг бөгөөд үүнийг програмдаа тоолуур болгон хялбархан нэгтгэх боломжийг олгодог. POST хүсэлт нь эхний/сүүлийн N жетоныг буцаах гэх мэт илүү олон параметр, функцуудыг дэмждэг.
Хүсэлт болгонд би токенжуулж чадах хамгийн дээд урт нь хэд вэ?
Та хүсэлт бүрт 64 мянга хүртэлх тэмдэгт илгээх боломжтой.
Хэсэглэх функц хэрхэн ажилладаг вэ? Энэ нь семантик бүлэглэлт мөн үү?
Хэсэглэх онцлог нь нийтлэг бүтцийн дохиололд тулгуурлан урт баримтуудыг жижиг хэсгүүдэд хувааж, текстийг утга учиртай хэсгүүдэд үнэн зөв сегментлэх боломжийг олгодог. Үндсэндээ энэ нь өгүүлбэрийн төгсгөл, догол мөрийн завсарлага, цэг таслал, зарим холбоо үг зэрэг утгын хил хязгаартай ихэвчлэн нийцдэг синтаксик шинж чанарт тулгуурлан текстийг сегментчилдэг (том!) регекс загвар юм. Энэ нь семантик бүлэглэл биш юм. Энэ (том) regex нь ердийн илэрхийллийн хязгаарлалтын хүрээнд байж болохуйц хүчтэй юм. Энэ нь нарийн төвөгтэй байдал, гүйцэтгэлийг тэнцвэржүүлдэг. Жинхэнэ семантик ойлголтыг regex-ийн тусламжтайгаар хийх боломжгүй ч энэ нь нийтлэг бүтцийн дохиогоор контекстийг сайн ойртуулдаг.
Segmenter API дахь 'endoftext' гэх мэт тусгай жетонуудыг та хэрхэн зохицуулах вэ?
Хэрэв оролтод тусгай жетон байгаа бол манай Segmenter API тэдгээрийг "special_tokens" талбарт оруулна. Энэ нь танд тэдгээрийг хялбархан таньж, өөрийн даалгаврын дагуу зохицуулах боломжийг олгоно, жишээлбэл. Тарилгын халдлагаас урьдчилан сэргийлэхийн тулд текстийг LLM-д оруулахаас өмнө тэдгээрийг устгана.
chunking англи хэлнээс бусад хэлийг дэмждэг үү?
Барууны хэлнээс гадна, chunking нь Хятад, Япон, Солонгос хэлтэй сайн ажилладаг.
